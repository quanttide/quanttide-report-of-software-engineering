从整个产品的演进线来看，AI 在探索一个核心问题：

**"规则引擎的判断，怎么用最小的 AI 成本让它变得可信？"**

这不是在做一个"AI 代码审查助手"——那太泛了。它是在精确定位 AI 在代码质量工具中**不可替代的最小角色**。

---

### 探索路径

```
阶段 1（已实现）：纯规则引擎
  ├── 确定：相同代码 = 相同结果
  └── 痛点：假阳性，用户不信任

阶段 2（实验 1/2）：规则 → 全部交给 LLM
  ├── 发现：34.8% 的 LLM 调用是浪费的
  └── 发现：LLM 的驳回理由高度重复（"测试函数"、"骨架文件"）

阶段 3（实验 3）：规则 → 启发式过滤 → LLM
  ├── 验证：7/8 的 LLM 驳回可以用 3 行 if 替代
  └── 验证：LLM 调用量降低 48%，驳回率 0%

阶段 4（下一步，报告末尾）：
  规则 → 启发式过滤 → 证据链组织 → LLM
  探索：不是"要不要 LLM"，而是"给 LLM 看什么证据，它才能做最准的判断"
```

---

### 它在探索的那个"边界"

```
问题类型                      谁处理         确信度
────────────────────────────────────────────────────
函数超过 30 行 → finding      规则引擎        100%
测试文件超过 30 行 → 跳过      启发式规则        87%
__init__.py 没测试 → 跳过      启发式规则        95%
57 行的函数但逻辑集中？          LLM            不确信但可判断
跨文件重构引入的新 bad smell？   LLM + 证据链    正在探索
```

AI 在找那条 **"规则定不了、启发式也猜不准、必须靠理解语义才能判断"** 的线在哪里。报告已经证明这条线确实存在（`scan_single_submodule` 案例），而且很窄——87.5% 的"假阳性"根本不需要 AI。

---

### 一句话

**它在探索：在不引入 LLM 不确定性和成本的前提下，规则引擎的边界能推到多远；当推到尽头后，LLM 最少需要看什么信息才能补齐剩下的缺口。**

从产品经理视角看，做这件事的理由是三层递进的。

---

### 第一层：用户信任

纯规则引擎的致命问题是**假阳性噪音**。

用户运行 `review`，看到 23 个 finding，点进去一看——8 个是测试函数太长、4 个是 `__init__.py` 没测试。几次之后，用户就不信这个工具了。"它报的东西我都要扫一遍再判断，那我还要它干嘛？"

**假阳性不解决，工具在用户心里的信用是负的。** 用户不是不用，是用了还要花更多精力去过滤——比不用更糟。

---

### 第二层：LLM 不是答案本身

直觉方案：上 LLM，让它判断每个 finding 是真问题还是假阳性。

但实验 2 证明了：**34.8% 的 LLM 调用是浪费的。** 每次调用都在回答同一个问题："测试函数太长算不算坏味道？"——而这个问题，一行 `if` 就能回答。

做这件事不是因为 LLM 不够好，而是因为**把模式化问题交给 LLM 是一种架构上的懒惰**。对产品来说，每次 LLM 调用都是成本、延迟和不确定性。用户等 3 秒得到"驳回，理由是测试函数"——这 3 秒不值得。

---

### 第三层：产品形态的分水岭

**这是决定"工具依赖 AI"还是"工具使用 AI"的分水岭。**

- **依赖 AI**：所有判断都走 LLM，离线/断网/高延迟时功能瘫痪
- **使用 AI**：规则引擎做 80% 的工作，启发式做 15%，LLM 只处理最后 5% 的边界情况

离线也能用、低成本、可复现、结果稳定——这些特征是"代码审查工具"这个品类的基本要求。任何必须联网才能做的代码审查工具，在 CI 场景下都是不可接受的。

**做这件事，是在确保 AI 成为产品的增强层，而不是基础设施。** 如果哪天 LLM API 涨价了、停服了、延迟变高了，产品不能跟着崩。

---

### 一句话

**不是因为 AI 做不了，而是因为大部分问题不需要 AI——把 AI 留给真正需要它的 12.5%，让产品在离线、低成本、可复现和智能之间找到可交付的平衡点。**
